### 1. マジックナンバーの弊害

**マジックナンバーとは** 
オブジェクトや配列の中身に使われる「名前のない生の数字や文字列」のこと。（例：`2` や `"jpg"`）

**なぜダメなのか**

- **意味不明**：値が何を表しているのかわからず、どのグループ（種別）に属するデータなのか推理が必要になる。
- **管理不能**：修正が必要になった際、どこを直すべきか判断できなくなる。

### 2 定数管理を活用する

**解決策（オブジェクトと配列の使い分け）** 

これらを役割ごとの「種別（グループ）」に分けて管理します。

- **オブジェクト**：特定のデータに「名前（ラベル）」を付けて呼び出すときに使う。
- **配列**：複数のデータを「一覧（リスト）」として一括で扱うときに使う。

### 3. 具体例

#### 3-1. オブジェクトの場合

■ 【**整理前】1つの箱に混ざっている**

中身が増えてくると、どのキーが何のための設定かパッと見で判断しにくくなります。

```
// ❌ 混ぜこぜの状態（マジックナンバーも残っている）
const APP_DATA = {
  adminRole: 1,      // ユーザーの種類？
  guestRole: 2,      // ユーザーの種類？
  maxLoginRetry: 5,  // システムの制限？
  timeoutSeconds: 30 // システムの制限？
};
```

 ■ 【**整理後】種別ごとに2つのオブジェクトに分ける**

「何についてのデータか」という**種別（グループ**）で箱を分け、さらに値に名前を付けます。

**グループA：ユーザー権限（USER_ROLE）**

```
const USER_ROLE = {
  ADMIN: 1,
  GUEST: 2
};
```

**グループB：システム設定（SYSTEM_CONFIG）**

```
const SYSTEM_CONFIG = {
  MAX_LOGIN_RETRY: 5,
  TIMEOUT_SECONDS: 30
};
```

■ **分けた後の使い方（筋道）**

分けることで、コードを読むだけで「どの種別の、何の設定か」がくっきりします。

```
// ⭕️ 種別が分かれているので読みやすい
if (user.role === USER_ROLE.ADMIN) {
  // 管理者の処理だとすぐわかる
}

if (retryCount > SYSTEM_CONFIG.MAX_LOGIN_RETRY) {
  // システムの制限値を超えたとすぐわかる
}
```

#### 3-2. 配列の場合

■ **【整理前】1つの配列に混ざっている**

中身がただの「文字列や数字のリスト」だと、どの要素が何の役割か分かりません。

```
// ❌ 混ぜこぜの状態
// 「許可するファイル」と「除外するファイル」が一緒で、どれがどれだか謎
const FILES = ["jpg", "png", "exe", "sh"]; 

// 使うとき：FILES[2]はダメなやつだっけ？と推理が必要
```

■ **【整理後】種別ごとに2つの配列に分ける**

「用途」という**種別（グループ**）で配列を分け、名前に役割を明示します。

**グループA：許可リスト（ALLOWED_FORMATS）**

```
// ⭕️ 「使っていいもの」だけのグループ
const ALLOWED_FORMATS = ["jpg", "png", "webp"];
```

**グループB：禁止リスト（DENIED_FORMATS）**

```
// ⭕️ 「実行してはいけないもの」だけのグループ
const DENIED_FORMATS = ["exe", "sh", "bat"];
```

■ **分けた後の使い方**

「名前（ラベル）がついた箱」を使い分けることで、コードの意図がくっきりします。

```
// ⭕️ 意味がはっきりする
if (ALLOWED_FORMATS.includes(uploadFile)) {
  // 許可された形式ならアップロードOK
}

if (DENIED_FORMATS.includes(uploadFile)) {
  // 禁止された形式なら警告を出す
}
```

### 3. 結論

オブジェクトでも配列でも、**「役割ごとに種別分けして、名前付きの箱に入れる**」ことが、整理整頓（セッティングの整理）を完成させる唯一の道です。
