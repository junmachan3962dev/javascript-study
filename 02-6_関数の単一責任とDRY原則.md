#### **1. 単一責任の原則（SRP）**

**「1つの関数には、1つの仕事だけをさせる**」というルール。

- **状態**：
  料理なら「切るだけ」「焼くだけ」と工程を分けること。

- **メリット**：
    - 中身が単純なので、読みやすく、テストしやすい。
    - 修正時に、他の仕事にバグが飛び火しない。

#### **2. DRY原則（Don't Repeat Yourself）**

**「同じコードを二度と書かない（繰り返さない）**」というルール。

- **状態**：
  同じ計算式や処理があちこちに出てきたら、1箇所にまとめること。

- **メリット**：
    - 修正が必要なとき、1箇所直せばすべてに反映される（修正漏れがない）。
    - コード全体の量が減り、スッキリする。

#### **3. 二つの関係性（なぜセットなのか）**

**「単一責任を守るからこそ、DRY（再利用）が可能になる**」という関係です。

**■ 悪い例（仕事が混ざっている）**

```
// 「消費税計算」と「画面表示」の2つの仕事をしている
function showTaxPrice(price) {
  const taxPrice = price * 1.1; // 計算
  console.log(`価格は${taxPrice}円です`); // 表示
}
```
- **問題点**：「計算結果だけ使いたい」時でも表示がついてくる。結局、別の場所でまた `price * 1.1` と書くことになり、**DRYが壊れる**。

**■ 良い例（仕事を分けて部品化する）**

```
// 仕事1：計算だけをする（単一責任）
const calcTax = (price) => price * 1.1;

// 仕事2：表示だけをする（単一責任）
const displayPrice = (amount) => console.log(`価格は${amount}円です`);
```
- **解決策**：`calcTax` はどこでも使い回せる「便利な部品」になった。同じ計算式を書かずに済むため、**DRYが守られる**。

#### **4. 結論**

- **単一責任**は、関数を使いやすい「部品」にすること。

- **DRY**は、その「部品」を使い回して、効率化すること。

**「仕事を1つに絞る。だから、どこでも使い回せる。」** これが整理されたコードの基本です。
